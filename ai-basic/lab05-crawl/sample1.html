    • 작성자 : 노경훈, 최근 변경 : 정승원 - 2025-09-16

    AI사업개발

    구분

    전주 실적(2025. 9. 8  ~ 2025. 9. 12 )


    [K-intelligence AI Agent 구축 프로젝트]


    주요

    업무

    현황


    • My K 오픈 준비 일정


    • My K 운영 계획

      • 에이전트 분과 : Agent는 연말까지 A-Engineering담당에서 내재화 형태로 운영 예정(통신 Agent는 외부 협력사 협업 형태)으로
        인수인계 설명회가 아닌 앱운영 위한 주요 내용 설명회로 진행

        • 9월 25일 출시 기준, 운영협력사 계약 10월 1일 목표로 진행 중

          • (참고) 인수인계 대면 설명회 일정

            • 1안) 계약 직후 : 10월 1일(목)~10일(금) 

            • 2안) 추석 직후 : 10월 13일(월)~15일(수)

    • My K 향후 일정

      • 상용 출시 목표일 : 9월 25일(목요일) (단, 10월 말 이후로 오픈일정 밀릴 것으로 예상됨)

        • 일반고객(200명) 및 임직원(1천명 예정) 대상 고객 테스트(서프본부, 9.12)

        • 9/9(월) ~ 9/11(목): 5차 Agent 검증(Device사업본부)

        • 9/15(월) ~ 9/18(목) : 5차 앱 검증(Device사업본부)

        ㅇ 상용 출시 전 품질 이슈 해소를 위한 긴급 진행 사항

        • (9.12) 세 가지 검증 결과 상세 리뷰 및 품질 수립 협의

        • (9.17) 품질 수립 기준에 따른 검증 결과 분석

        • (9.18)  검증 결과 검토 및 품질 현황 확인 담당급 미팅


    • 하반기 에이전트 진행 시 개선 요청 사항 및 개선 방향

      내용

      유관부서

      개발총괄 PM 역할 필요

      • 예를들면, Agent 개발팀과 앱 개발팀이 분리되어 있는데, 총괄 개발PM 역할이 없어 커뮤니케이션 및 의사결정이 어려웠음

      P-PMO팀, Custom Agent팀,
      P-서비스개발팀

      [개발 프로젝트 관리]

      WBS 기반으로 개발 프로젝트 관리가 되어야 함

      개발 부서

      ㅇ 업무 시나리오 정의 시 Agent에 최적화된 프로세스로 검토 : 현재 구현되어 있는 웹화면을 1대1로 옮기기 보다는 Agent에 최적화된 프로세스 검토

      ㅇ 사전 기획 완료 후 설계 및 개발착수

      ㅇ API는 Agent 개발부서에서 필요한 Input/Output 정의 후 제공 시스템으로 요청

      사업팀/기술전략TF, A-기획팀, Custom Agent팀,
      연동 API 개발부서

      [시나리오 작성]

      A기획팀에서 작성한 시나리오가 결국 피그마로 그려지고 피그마 파일이 사업부서, 개발부서들이 검토하는 자료로 거의 활용이 되었음.

      시나리오 작성단계에서 각각의 발화들이 별도로 관리되는게 효율적일 것 같아 발화내용을 별도 액셀로 관리하면 좋을거 같음

      사업팀/기술전략TF, A-기획팀

      [화면기획서]

      (피그마 작업시) 일부 페이지만 작업하는게 아닌 전체 시나리오별 기획사항을 작업하여 지속 업데이트 관리하는 방식으로 진행

      최종 산출물은 피그마로 전체 Flow를 점검할 수 있도록 진행

      시나리오 기획서를 기반으로 U.I기획이 필요한 화면만 피그마로 정리, 전체 Flow별 검토관점에서 여러 파일을 동시에 점검해야 하는 상황이 반복적으로 발생하여 점검 시간 지속 증가.


      문서별 sync 불일치 → 기획서와 화면설계서 통합 필요

      사업팀/기술전략TF, A-기획팀, X-서비스기획팀

      [API 개발]

      현재 KT닷컴 Shop은 API구조로 설계되어 있지 않음.

      제휴 연동 및 필요시 분석을 통해 연동에 필요한 규격으로 별도 개발을 통해 진행 중.

      하반기 신규과제는 시나리오 및 화면기획을 기반으로 에이전트에 필요한 API에 대한 선 분석 후 (Shop 개발측에) API 개발 진행되도록 개선 필요

      짧은 일정으로 서비스에 필요한 API를 분석 후 API를 선개발 후 에이전트 개발 시 기 개발된 API에 대한 이슈 논의 및 재개발 반복.

      초기 예상 공수대비 추가 투입으로 예산 이슈 발생

      사업팀/기술전략TF, A-기획팀, Custom Agent팀,
      연동 API 개발부서

      [커뮤니케이션]

      사업부서, 에이전트개발, UI, 앱개발등 각 팀들이 각 업무 환경에서도 동일한 내용을 다른 표현으로 사용하고 있는 부분들이 있고

      그 표현으로 회의를 진행하다보니 소통이 간혹 잘 안되는 부분들이 있었던 것 같음

      예를 들면, ‘화면설계서’에 대해서도 시나리오,SB,등등 다른 표현으로 하다보니 사각지대가 발생했던 것 같음

      Shop화면에서 특정 영역을 지칭하는 부분도 조직별 표현이 달랐던 걸로 기억함

      (어떤 영역을 그대로 사용하자며 얘기한 부분이 서로가 이해하는 바가 달라서.. 감정들이 격앙되기도 했었고..)

      각자가 최대한 표준 용어, 혹은 이해시킬 수 있는 단어로 논의하는 그라운드 룰을 만들면 좋을 것 같음

      모든 부서


      구분 

      문제점

      개선방향

      협업 구조

      Agent 개발팀 vs App 개발팀으로 개발 파트가 나뉘어 있어 협의가 어려운 경우 발생

      모든 개발팀을 아우르고 기획팀과 커뮤니케이션 할 수 있는 총괄개발 PM 필요

      각 부서내 결정사항에 대한 협의/공유가 늦음

      부서내 결정사항에 대해 관련부서 협의 필수

      Slack 등 커뮤니케이션 도구 활용을 통해 빠른 공유

      문서

      작성/공유/관리방식

      기획 시 참고할 선행 문서나 레퍼런스 부족 

      기획서 작성 전 사업부서에 요청할 문서 정의/요청

      예) 사업목표, 서비스 시나리오, 서비스 사용자 현황(or 목표) 등

      Agent마다 조금씩 다른 기획서 양식

      Figma 공통 기획서 템플릿 마련

      버저닝 규칙 정의

      기획서와 화면설계서의 분리

      기획서와 화면설계서 통합

      빈번한 스펙 변화로 인한 개발 일정 압박 및 백로그 누적

      스펙 프리징 원칙 정의

      Figma, ppt 파일 용량이 커짐에 따라 이메일/confluence 등록이 어려움

      협력사 협업 시 보안문서해제의 번거로움 발생

      관련부서 Figma 접근권한 제공 / 문서 공유방식 일원화

      X-서비스기획팀 통해 추가 권한 문의

      기획팀이 프롬프트 내용을 알 수 없어 수정 요청이 어려움

      마스터/에이전트별 프롬프트를 공유해 협업 효율성 제고

      추후검토

      테스트 셋 공유 및 인풋/아웃풋 검증 가능한 테스트 환경 구축
